# 网站集成与运行安全专项计划

> 文件路径：`docs/integration_security_delivery_plan.md`
>
> 文件定位：本文件是“网站集成与运行安全”的专项执行计划，不是项目唯一当前状态源。
>
> 唯一当前状态源仍为：`docs/project_master_roadmap.md`。
>
> 任何阶段开始、完成、延期、取消或顺序调整，都必须同步更新 `docs/project_master_roadmap.md`；本文件负责保存专项范围、顺序、门禁和延期事项，避免后续对话遗漏。

---

## 1. 已冻结的总体决策

当前阶段继续以核心功能实现为主，不因运行安全扩展中断 F6-2、Vanna 解耦、多数据源、路由和报表主线。

网站接入分为两种性质：

1. **测试环境集成**：可以在完整安全扩展前进行，用于验证接口契约、身份上下文、SSE、会话、文件与页面嵌入。
2. **真实网站正式接入**：必须在最低运行安全门禁完成后进行，不得直接向真实用户开放。

冻结顺序：

```text
当前功能阶段
→ Vanna 解耦与项目瘦身
→ 多数据源核心
→ PostgreSQL 迁移适配
→ MySQL 接入
→ 数据源路由
→ 报表能力
→ 网站测试环境集成
→ 运行安全与集成验收
→ 真实网站正式接入
→ 深层安全与生产优化
```

## 2. 阶段划分

### 阶段 A：核心功能完成

当前只推进功能主线，不插入大范围安全重构。

顺序：

```text
A1  完成 F6-2
A2  Vanna 解耦与项目瘦身
A3  多数据源核心抽象
A4  PostgreSQL 迁移适配
A5  MySQL 接入
A6  数据源路由
A7  报表能力
```

阶段 A 原则：

- 只处理当前阶段明确目标；
- 不提前实现网站权限、频控、字段脱敏等完整体系；
- 发现安全问题时记录到本文件，不在未授权情况下扩大当前修改范围；
- 不因专项计划存在而打断 F6-2 当前验收流程。

### 阶段 B：网站测试环境集成

该阶段只允许测试环境或隔离环境，不得面向真实生产用户开放。

范围：

```text
B1  固定后端模块接口契约
B2  固定用户、部门、角色和请求上下文透传格式
B3  固定 SSE 事件契约
B4  固定会话标识与历史恢复契约
B5  固定文件、CSV、图片和报表下载契约
B6  完成目标网站测试环境嵌入
B7  记录目标网站已有的鉴权、频控、审计和反向代理能力
```

阶段 B 目标不是上线，而是通过真实集成场景确定：

- 用户身份从哪里产生；
- 用户 ID、部门和角色如何传递；
- 真实 IP 如何获取；
- 网站是否已有统一频控；
- 网站是否已有审计平台；
- 数据权限由网站、Agent 还是数据库共同控制；
- SSE、文件下载和会话保存的最终边界。

阶段 B 完成后，必须根据实际集成结果修订阶段 C 的实现细节。

### 阶段 C：运行安全与集成验收

阶段 C 是真实网站正式接入前的强制门禁。

#### C1：数据库异步隔离与并发控制

目标：同步 `psycopg2` 查询不得阻塞 FastAPI 主事件循环。

要求：

- SQL 执行放入专用 `ThreadPoolExecutor` 或等价的受控线程执行器；
- 不使用无边界线程创建；
- 使用 `asyncio.Semaphore` 或等价机制限制数据库并发；
- 保留数据库连接超时、`statement_timeout`、`lock_timeout` 和只读事务；
- 验证单个慢查询不会阻塞其他健康检查或普通请求；
- 增加并发与超时回归测试。

#### C2：资源限制与接口频控

目标：防止大结果、重复点击、并发追问或高成本 SQL 打满服务和数据库。

要求：

- 明细查询强制结果行数上限；
- 聚合查询不能仅依赖 `LIMIT` 作为扫描保护；
- 设置返回结果体积上限；
- 设置单用户或单 IP 请求频控；
- 设置单用户并发上限；
- 防止短时间重复提交相同请求；
- 根据数据库能力配置 `work_mem`、`temp_file_limit` 等资源边界；
- 保留并验证 `statement_timeout`；
- 后续是否引入 `EXPLAIN` 成本门禁，必须依据实际慢查询证据决定。

说明：PostgreSQL 没有可直接通用设置的 `max_rows_examined`。不得把简单追加 `LIMIT` 描述为完整扫描保护。

#### C3：数据分级与出口控制

目标：真实用户只能获得允许暴露的数据。

先建立配置化字段分级：

```text
DENY       禁止返回
MASK       掩码后返回
REDUCE     降低精度后返回
ALLOW      正常返回
```

出口控制必须发生在：

```text
数据库执行成功
→ DataFrame 对外构造前
→ CSV 或文件写入前
→ LLM 结果上下文前
→ SSE 返回前
```

要求：

- 不只在前端遮挡；
- 坐标字段是否截断必须由数据分级决定，不统一盲目处理；
- 站点名称、告警阈值、坐标、内部编码等字段逐项盘点；
- 拒绝字段不得出现在 CSV、Trace 或最终响应中；
- 日志和证据分支也必须执行脱敏。

#### C4：结构化审计与可观测性

目标：每个真实请求均可追踪、解释和回放。

最低记录：

```text
trace_id
request_id
conversation_id
user_id / department / role
原始问题
最终执行 SQL
SQLGuard 结果
SQL 执行耗时
总请求耗时
结果行数与列名
结果预览的受控摘要
错误类型
LLM 调用次数
输入 / 输出 / 总 Token
模型与 Provider
```

要求：

- 敏感字段和凭据不得进入日志；
- 定义日志保留周期、轮转和删除机制；
- Provider 超时、数据库错误、SQLGuard 阻断和用户取消必须有独立状态；
- O1/O3 请求级 Trace 在正式提交后作为本阶段基础，不重复从零实现；
- 诊断 Trace 与长期运营审计分开，避免永久保存完整 Prompt 和数据。

#### C5：Memory 检索评估扩展

目标：提高检索可解释性，不再只看最终 SQL 是否偶然正确。

要求：

- 对固定回归题输出检索到的 Memory 顺序、类型、分数、过滤原因和实际注入项；
- 保留现有 HTTP 与 Memory 回归；
- 增加同义问法、粒度冲突和易混表测试；
- 对错误 SQL 记录目标表是否进入候选、正确 SQL 示例是否进入检索和注入；
- 先用评估证明问题，再决定是否调整阈值或存储结构。

#### C6：是否拆分 Chroma Collection

该项不是默认必须执行。

只有在 C5 证明不同 Memory 类型确实产生稳定干扰后，才允许设计拆分方案。

要求：

- 先盘点 Vanna 2.0.2 实际 Collection 和 Metadata 结构；
- 不得仅依据“共用一个持久化目录”认定所有类型共用一个检索阈值；
- 拆分必须提供迁移、回滚、记录数、指纹和回归方案；
- 无证据时保持当前存储结构。

## 3. 真实网站正式接入门禁

进入真实网站正式接入前，至少必须满足：

```text
C1 数据库异步隔离与并发控制：通过
C2 资源限制与接口频控：通过
C3 数据分级与出口控制：通过
C4 结构化审计与可观测性：通过
C5 固定检索评估基线：通过
```

C6 是否拆分 Collection 为证据驱动决策，不是强制前置条件。

同时必须满足：

- 核心功能回归通过；
- 网站测试环境集成通过；
- 身份上下文可追踪；
- 数据库使用只读角色；
- 正式资产有备份和回滚点；
- 真实网站集成验收报告已提交；
- 不存在仅凭 Codex 汇报、未读取原始证据便宣布通过的阶段。

## 4. 阶段 D：真实网站正式接入

范围：

```text
D1  正式身份和部门上下文接入
D2  正式数据源配置
D3  正式频控和并发策略启用
D4  正式数据分级策略启用
D5  正式审计日志启用
D6  灰度开放
D7  正式验收
```

要求：

- 先灰度，不一次性向所有用户开放；
- 预设回滚开关；
- 监控慢查询、错误率、Token、并发和数据库负载；
- 灰度期间发现的问题优先修复，不同时扩展新功能。

---

## 5. 阶段 E：深层安全与生产优化

可以在真实集成稳定后按证据推进：

- 部门级、角色级和字段级精细权限；
- SQL 执行计划成本门禁；
- 动态配额；
- 集中日志、指标和告警平台；
- 容量测试和压力测试；
- 高可用与故障转移；
- 数据源级隔离；
- Prompt、Memory 和模型调用的长期质量分析；
- 根据 C5 结果决定 Collection 重构；
- 生产部署和运维自动化。

## 6. 状态维护规则

### 状态标记

每项只允许使用：

```text
NOT_STARTED
IN_PROGRESS
BLOCKED
COMPLETED
DEFERRED
CANCELLED
```

### 更新要求

每次 Codex 完成一个阶段后，必须：

1. 更新本文件对应项目的状态、证据路径和结论；
2. 同步更新 `docs/project_master_roadmap.md` 的当前阶段；
3. 不删除未完成或延期事项；
4. 若调整阶段顺序，写明原因和批准来源；
5. 若发现额外问题，只记录，不自动扩大当前阶段；
6. 本地运行证据必须脱敏后提交到独立 `evidence/*` 临时分支；
7. 临时证据分支不得合并到 `master`；
8. ChatGPT 必须读取原始证据后，才能将验收状态更新为 `COMPLETED`。

### 审查可信度标记

每个验收结论应标记：

```text
CODEX_REPORTED
DIRECTLY_VERIFIED_BY_CHATGPT
NOT_VERIFIED
```

只有 `DIRECTLY_VERIFIED_BY_CHATGPT` 才能作为正式阶段通过依据。

## 7. 跨 ChatGPT 窗口交接规则

下一任 ChatGPT 窗口开始项目工作时，必须先指挥 Codex：

```text
1. 读取 docs/project_master_roadmap.md
2. 读取 docs/integration_security_delivery_plan.md
3. 输出两个文件中的当前阶段、禁止越界项和下一步
4. 核对 Git HEAD、origin/master、工作区和正式资产状态
5. 只执行当前阶段，不提前开展后续安全或网站集成任务
```

每次准备结束窗口或生成新窗口交接提示词时，必须包含：

```text
- 本专项计划文件路径
- 当前专项阶段和状态
- 下一个允许执行的最小步骤
- 延期安全事项清单
- 证据分支列表和保留规则
- 要求 Codex 在阶段变化后维护本文件
```

若 Codex 修改本文件，ChatGPT 必须审查差异，确认：

- 没有把本文件变成第二个当前状态源；
- 没有擅自删除延期事项；
- 没有提前宣布真实网站可上线；
- 阶段状态与总路线一致；
- 本地运行结论存在可直接审查的证据分支。
