# 训练数据交付 SOP

## 定位与边界

本 SOP 是本项目的训练治理流程，不是 Vanna 官方强制流程。Vanna 2.0.2 的 `Agent` 不会在 SQL 工具成功后自动保存该次 SQL；正式训练写入应显式调用 `AgentMemory.save_tool_usage()`。当前线上服务没有注册 Memory 写入工具，因此线上问答不会形成自动训练写入。

代码提交和 Chroma 数据交付是两个独立验收对象：代码以 Git 提交和测试结果验收，Chroma 以备份、逐文件指纹、写后查询和交付清单验收。代码提交成功不能替代 Chroma 数据验收，Chroma 数据变化也不得混入代码提交。

每次实际执行只能进入一个明确阶段。禁止用单个命令或无人值守流程自动连续执行 T0—T8；进入下一阶段前必须验收上一阶段证据并由负责人明确授权。本文件不保存真实密码、API Key、个人信息或绝对备份路径。

## T0：训练范围冻结

- 建立符合 `schema_version=1.0` 的批次 JSON，固定批次编号、训练级别、来源、样本和预期新增数量。
- 所有样本必须标记为 `approved`，并记录审查原因、预期行为及预期表。
- 计算确定性 `batch_content_sha256`，冻结后不再原地修改；变更应形成新批次。
- 本阶段只保证批次内 `sample_id` 唯一，不声称已经验证正式 Chroma 中的全局唯一性。

## T1：静态训练审查

- 使用 `tools/validate_training_batch.py` 校验 JSON 契约。
- 每条 SQL 必须调用项目现有 `SQLGuard.validate(sql, query)`，不得复制或弱化规则。
- 校验单语句、SQLGuard 结论以及 `used_tables` 与 `expected_tables` 的一致性。
- 输出可复核的控制台结果；确需文件时只能写到仓库外的临时或交付目录。
- T1 只读，不连接数据库、不打开 Chroma、不写入 Memory。

## T2：隔离预演

- 仅在仓库外临时 clone 和隔离数据目录中预演。
- 应用冻结批次和待交付代码，确认环境变量、输入摘要、输出位置及失败行为。
- 不得把临时结果、缓存或隔离数据提交到正式仓库。

## T3：训练前回归

- 执行批准的后端回归清单，记录脚本数量、通过数、失败数和提交号。
- 任一安全边界或既有回归失败时停止，不得进入正式写入。
- 再次核对批次摘要与 T0 冻结值一致。

## T4：正式备份与指纹

- 对正式 Chroma 数据目录执行受控备份，并记录逐文件相对路径、大小和 SHA-256。
- 备份位置由执行环境安全配置，不写入本文、批次 JSON 或代码仓库。
- 验证备份可读、指纹清单完整后才能批准 T5。

## T5：正式训练写入

- 只处理一个已批准且摘要一致的批次。
- 使用 Vanna 2.0.2 `AgentMemory.save_tool_usage()` 写入批准的 `run_sql` 工具用法。
- 写入数量必须与 `expected_new_memory_count` 一致；不得同时执行其他训练批次或代码变更。
- 出现异常立即停止后续写入并保留完整执行证据。

## T6：写后 Memory 验证

- 通过正式 Memory API 检索本批次样本，核对问题、工具、参数、训练级别和数量。
- 检查批次样本是否重复、缺失或出现非批准内容。
- 重新计算正式数据目录逐文件指纹，形成 Chroma 数据交付差异清单。

## T7：训练后回归

- 重新执行训练前同一回归清单，并加入本批次目标问题的受控检索验证。
- 对比 T3 与 T7，确认无安全、检索和问答质量回退。
- 不得用一次人工成功查询代替完整回归结果。

## T8：交付验收

- 代码对象：提交号、允许文件清单、测试摘要、远端推送状态。
- Chroma 对象：备份标识、写前写后指纹、预期与实际新增数量、Memory 验证结果。
- 两类对象分别签收；任何一个缺少证据，交付均不得标记完成。
- 归档批次 JSON、确定性摘要和各阶段审批记录，但不归档密钥或个人信息。

## 异常回滚

- T0—T4 失败：不进入 T5，修正后创建或重新冻结批次并重新审查。
- T5 写入中失败：停止写入，禁止用未审查脚本补写；记录已写数量和失败位置。
- T6 或 T7 失败：依据 T4 已验证备份恢复正式 Chroma，再逐文件核对恢复后指纹。
- 回滚只处理本次训练数据对象，不回退无关代码或其他批次；代码问题使用独立 Git 修复流程。
- 恢复和验证完成前，不得继续后续批次或宣称交付成功。
