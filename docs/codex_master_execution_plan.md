# Codex 项目自主执行总计划

## 1. 文件定位

本文件是 Codex 长期执行本项目的唯一操作计划。

Codex 每次开始工作时必须先读取本文件，再读取：

```text
docs/project_master_roadmap.md
```

其中：

* 本文件负责执行方法、总目标、阶段顺序和自主决策规则；
* `docs/project_master_roadmap.md` 负责当前状态、正式资产基线和历史验收事实；
* `docs/training_master_route.md` 只作为训练规范与历史参考；
* `docs/integration_security_delivery_plan.md` 只作为网站集成和运行安全的专项参考；
* `docs/regression_baseline.md` 中与当前 Runner、Suite 或正式资产不一致的内容均视为历史记录，不得覆盖当前代码和总路线。

不得再创建第二套总计划。

---

## 2. 项目最终目标

将当前 PostgreSQL 中文数据问答系统逐步建设为可长期维护、支持多数据源、能够生成报表并可接入外部网站的生产级数据 Agent。

最终能力包括：

```text
自然语言问题
→ 正确识别数据源
→ 检索对应 Metadata、DDL 和 SQL 示例
→ 生成正确 SQL
→ SQLGuard 校验
→ 只读执行数据库查询
→ 返回可信答案
→ 生成图表、表格、仪表板或完整报表
→ 保存和导出
→ 通过统一 API 接入外部网站
→ 支持真实用户、权限、审计和运行安全
```

## 4. 核心执行原则

### 4.1 功能主干优先

任何工作必须直接服务于当前阶段的可用结果。

不得因为以下事项阻断功能主线：

```text
证据形式还可以更完善
日志字段还可以更多
摘要还可以更精确
极端异常路径未覆盖
代码还能进一步抽象
目录还能进一步美化
可能存在但尚未出现的理论风险
```

只有以下问题允许阻断：

```text
会造成正
式数据丢失或不可恢复
会误写、误删正式资产
会导致当前功能结果错误
缺少必须由用户提供的业务决策或外部凭据
当前环境确实无法继续执行
```

### 4.2 验收标准冻结

每个阶段开始时确定验收标准。

执行开始后：

* 不得增加新的硬门禁；
* 不得因为发现额外优化点而提高验收线；
* 不影响当前功能的问题记入技术债；
* 技术债不得立即衍生为新阶段。

### 4.3 禁止阶段树状膨胀

不得创建类似：

```text
O5
→ O5-C
→ O5-C-P1
→ O5-C-P2
→ O5-C-R1
→ O5-C-R2
```

每个阶段最多包含：

```text
第一次实现或执行
一次针对真实失败的修复
一次复验
```

超过三轮仍未完成时，停止继续增加子阶段，重新检查：

```text
原验收条件是否错误
环境设计是否错误
是否正在解决非主目标问题
```

### 4.4 环境问题在原阶段内解决

以下问题属于执行环境调整，不得单独建立新阶段：

```text
虚拟环境路径错误
detached worktree 缺少被 Git 忽略的依赖
端口占用
cwd 错误
子进程 executable 错误
环境变量未继承
临时目录不存在
Windows 路径或换行差异
```

环境调整不修改业务功能时，Codex应当：

```text
确认根因
选择可工作的安全执行方式
在原阶段继续
```

不得为了环境问题生成新的路线、计划书或治理阶段。

## 6. 每个阶段的固定工作方式

每个阶段只执行以下六步：

```text
1. 读取当前状态
2. 检查运行环境
3. 实现当前目标
4. 执行冻结测试
5. 提交代码和更新状态
6. 输出简明结果
```

### 6.1 开始前检查

必须确认：

```text
git status
HEAD
origin/master
Python 实际路径
Python 是否存在
源码实际 PROJECT_ROOT
子进程 cwd
子进程 executable
端口状态
正式资产路径
测试副本路径
```

若使用 detached worktree，必须先确认：

```text
虚拟环境是否在工作树中
被 .gitignore 排除的运行依赖是否存在
Runner 是否根据 __file__ 重算环境路径
```

不能确认时，优先在干净的主工作区运行代码，并继续使用独立数据副本保证资产隔离。

### 6.2 修改边界

只修改当前阶段需要的文件。

发现其他问题时：

```text
记录问题
说明影响
不自动修改
```

### 6.3 测试

测试分为三档：

普通代码修改：

```text
相关单元测试
py_compile 或 TypeScript 检查
构建检查
```

阶段功能完成：

```text
目标功能测试
相关固定回归
```

正式资产写入或切换：

```text
完整备份
隔离副本验证
正式写入或目录切换
切换后完整回归
失败时恢复
```

不得把正式资产级证据要求应用到普通代码修改。

### 6.4 失败处理

测试失败时：

```text
定位唯一直接根因
只修复该根因
重新运行受影响测试
```

不得同时重构无关模块。

如果是环境失败：

```text
修复环境
继续原阶段
```

如果是单个回归用例失败：

```text
只分析该用例
不重新设计整个回归体系
```

### 6.5 提交

阶段通过后：

```text
检查 diff
只暂存授权文件
提交
推送 master
确认 HEAD = origin/master
确认工作区干净
```

普通阶段不创建 evidence 分支。

只有以下情况才需要独立 evidence 分支：

```text
正式 Chroma 写入或切换
正式 Metadata 替换
重大数据恢复
生产网站正式验收
难以仅靠 Git diff 和测试结果证明的高风险操作
```

### 6.6 输出格式

每次只输出：

```text
完成目标
修改文件
测试结果
提交 SHA
遗留技术债
下一阶段
```

不得输出几十项未被要求的治理字段。

## 8. 当前阶段：完成 F6-2

当前阶段目标：

```text
完成新的 PostgreSQL Metadata 和相应 DDL Memory 更新机制，
在不破坏当前正式能力的情况下完成候选验证、正式切换和最终回归。
```

### 8.1 当前剩余工作

按顺序自主完成：

```text
1. 在当前 managed-v1 正式资产隔离副本上执行 20 个 HTTP + 6 个 Memory 基线 ✅
2. 在候选 Metadata 和候选 Chroma 隔离副本上执行同一套完整验收 ✅
3. 比较旧正式与候选结果 ✅
4. 候选通过后，备份当前正式资产 ✅
5. 切换正式 Metadata 和正式 Chroma
6. 从切换后的正式资产创建全新副本
7. 再执行 20 个 HTTP + 6 个 Memory
8. 更新总路线和正式基线
9. 提交并清理工作区
```

### 8.2 当前环境阻断处理

当前 Runner Harness 会根据源码目录寻找：

```text
vanna_venv\Scripts\python.exe
```

detached worktree 不包含该虚拟环境。

处理方式固定为：

```text
从干净的主工作区执行 Runner
使用仓库外 validation_chroma
使用仓库外 validation_agent_data
不把正式 Chroma 作为 data-dir
```

该问题属于原阶段运行环境调整，不得创建新的修复阶段。

主工作区 Runner 复验发现 `level2_water_intake_route` 在精确命中 approved SQL 示例时仍自行删减输出字段。当前已在原阶段内修复：精确同题必须原样执行 approved SQL，语义近似问题仍只把示例作为参考。定向用例通过，随后旧正式隔离完整基线通过 `20/20 HTTP + 6/6 Memory`。

候选 Metadata 与候选 Chroma 的隔离完整验收已通过 `20/20 HTTP + 6/6 Memory`，与旧正式基线的 20 个用例通过判定完全一致。候选仅在小时水质用例中额外保留题目指定的 `station_id` 列，仍满足冻结契约。切换前正式资产已备份并完成指纹等价校验；下一步为正式 Metadata 与正式 Chroma 切换，执行前必须取得用户明确授权。

### 8.3 F6-2 完成标准

必须满足：

```text
旧正式隔离基线：20/20 HTTP，6/6 Memory
候选切换前验收：20/20 HTTP，6/6 Memory
正式切换成功
切换后验收：20/20 HTTP，6/6 Memory
正式资产前后事实清晰
代码和文档已提交
工作区干净
```

已知零行回答可能带有未经验证的原因推测或后续建议，记录为回答质量技术债，不阻断 F6-2。

## 10. 阶段 3：多数据源核心

目标：

```text
让数据库类型、Metadata、Memory、Runner 和 SQL 方言可独立配置。
```

核心结构：

```text
DataSourceConfig
DataSourceRegistry
BaseDataSourceAdapter
PostgreSQLAdapter
MemoryNamespace
AgentFactory
ChatService
```

第一版要求：

```text
用户或宿主显式选择 source_id
不让 LLM 自动猜数据库
不同数据源使用独立 Metadata
不同数据源使用独立 Chroma
不同数据源 SQL 不得互相执行
```

完成标准：

```text
同一个后端可注册多个数据源
PostgreSQL 功能保持不变
数据源之间 Metadata、Memory 和 SQL 完全隔离
```
