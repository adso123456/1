# Vanna 项目训练总路线

> 本文件用于统一项目训练阶段、交付流程和持续学习机制。后续 Codex、ChatGPT 与人工审查均以本文件为准，不得擅自改名、合并或跳过阶段。

## 一、总原则

本项目训练体系分为四个相互独立但前后衔接的层次：

1. 训练前安全基础设施
2. 四级正式训练
3. 每批训练的 T0—T8 交付流程
4. 四级训练完成后的在线持续学习

四者职责不同，禁止混用：

- 安全基础设施解决“能否安全训练”。
- 四级训练解决“要训练什么能力”。
- T0—T8 解决“一批训练数据如何安全交付”。
- 在线持续学习解决“上线后如何越用越聪明”。

---

## 二、训练前安全基础设施

该阶段不属于四级正式训练，只负责建立训练前置条件。

### 0A：运行与数据库安全基线

目标：

- 移除数据库硬编码账号密码。
- 数据库连接默认只读。
- 固定超时参数。
- 原始问题通过可信服务端链路传递。
- 禁用旧旁路入口。
- 保证现有回归测试全部通过。

当前状态：**已完成并验收通过**。

### 0B-1：训练批次契约、静态审查与批次指纹

目标：

- 定义版本化训练批次 JSON 契约。
- 使用现有 `SQLGuard` 静态审查 SQL。
- 生成确定性 `batch_content_sha256`。
- 建立 CLI 和稳定错误码。
- 保证不连接数据库、不打开 Chroma、不执行训练。

当前状态：**已完成并验收通过**。

### 0B-2：Chroma 只读审计、备份、指纹与恢复

目标：

- 只读清点正式 Chroma 内容。
- 建立逐文件大小与 SHA-256 指纹。
- 建立仓库外备份机制。
- 验证备份可读与可恢复。
- 建立正式 Chroma 与副本之间的隔离规则。

### 0B-3：隔离训练写入器、幂等与写后核验

目标：

- 实现受控的 `AgentMemory.save_tool_usage()` 写入器。
- 单批次、单方向写入。
- 写入前校验批次摘要。
- 写入后核对新增数量、内容和重复项。
- 防止重复执行产生重复 Memory。
- 禁止直接操作正式 Chroma 内部文件。

### 0B-4：Chroma 副本完整 SOP 演练

目标：

- 在仓库外临时 clone 和 Chroma 副本上完整执行 T0—T8。
- 使用少量模拟样本演练。
- 验证训练、核验、回归和回滚。
- 不写正式 Chroma。

只有 0B-4 验收通过后，才允许开始大规模正式训练。

## 三、四级正式训练

四级训练是项目固定的正式训练主线，不得另起名称替代。

### 第一级：107 张表结构与业务元数据训练

目标：让 Agent 对全部可训练数据库对象具备可检索的结构和业务知识。

必须覆盖：

- 数据库真实表清单。
- 表名、表注释和业务用途。
- 字段名、字段类型、字段注释和业务含义。
- 主键、外键和推荐 JOIN 路径。
- 时间字段、维度字段和指标字段。
- 单位、枚举值和聚合限制。
- 可加总字段与不可加总字段。
- 易混表、易混字段和口径差异。
- 允许表、限制表和禁止字段。
- 元数据检索索引。
- 必要的 Text Memory 业务规则。

验收标准：

```text
数据库确认可训练表数 = 107
知识卡覆盖数 = 107
元数据索引覆盖数 = 107
缺失表 = 0
重复表 = 0
未知表 = 0
无效字段引用 = 0
无效关系引用 = 0
禁止对象误开放 = 0
表定位检索测试通过
```

只有满足以上条件，才可以认定“107 张表基础知识已经学完”。

### 第二级：基础 SQL 查询训练

前提：第一级已能准确定位表、字段和基础业务含义。

训练内容：

- 字段选择。
- 条件过滤。
- 排序。
- 分页。
- 去重。
- 分组。
- 计数。
- 求和。
- 平均。
- 时间范围。
- Top N。
- 单表基础统计。

训练对象主要为 Tool Memory：

```text
用户问题
→ 正确表和字段
→ 正确基础 SQL
```

不要求机械地为 107 张表配置相同数量的 SQL，但必须覆盖实际可查询表的主要基础用法。

### 第三级：业务语义与复杂查询训练

项目现有 `P0 / P1 / P2` 属于本级内部优先级或难度分层，不是新的顶级训练等级。

训练内容：

- 口语化业务问法。
- 同义词和别名。
- 业务指标口径。
- 复杂筛选。
- 跨表 JOIN。
- 多层聚合。
- 排名与趋势。
- 同比与环比。
- 易混淆问题。
- 业务规则约束。

现有 P0、P1、P2 样本属于第三级既有训练资产，但不能替代第一级的 107 表全覆盖。

### 第四级：图表与可视化问答训练

前提：表定位、SQL 和业务口径已经正确。

训练内容：

- 何时使用表格。
- 何时使用柱状图、横向柱状图、折线图、面积图和饼图。
- 维度、指标、时间轴和 series 映射。
- 多序列图表。
- 单问题多图表。
- 图表与文字答案一致性。
- 数据不适合制图时拒绝强行制图。

本级不得用于弥补第一级、第二级或第三级的缺失。

## 四、每批训练固定执行 T0—T8

T0—T8 不是第五级训练，而是所有正式训练批次共用的交付流程。

### T0：训练范围冻结

- 冻结批次 JSON。
- 固定 `training_batch_id`、样本、训练等级、来源和预期新增数量。
- 计算完整批次摘要。

### T1：静态训练审查

- 校验 JSON 契约。
- 调用现有 `SQLGuard`。
- 校验表、字段、单语句和批准状态。
- 不连接数据库、不打开 Chroma。

### T2：隔离预演

- 在仓库外临时 clone 中执行。
- 使用隔离数据目录。
- 不写正式 Chroma。

### T3：训练前回归

- 执行批准的完整回归清单。
- 再次核对批次摘要。
- 任一安全或回归失败立即停止。

### T4：正式备份与指纹

- 备份正式 Chroma。
- 记录逐文件大小与 SHA-256。
- 验证备份可读。

### T5：正式训练写入

- 只执行一个已批准批次。
- 使用受控 Memory API 写入。
- 禁止并行执行其他训练批次。

### T6：写后 Memory 验证

- 核对实际新增数量。
- 检查重复、缺失和非批准内容。
- 生成写前写后差异清单。

### T7：训练后回归

- 重新执行与 T3 相同的回归。
- 增加本批次目标检索和问答验证。
- 对比训练前后结果。

### T8：交付验收

分别验收：

- 代码提交对象。
- Chroma 数据对象。
- 批次输入与摘要。
- 写前写后指纹。
- 回归结果。
- 回滚证据。

任何一个对象缺少证据，都不得标记交付完成。

## 五、在线持续学习

在线学习不属于第五级训练，而是四级训练完成后的持续学习机制。

目标：实现“越用越聪明”，但禁止无门禁自动污染正式 Memory。

线上问答产生的内容首先只是学习候选。候选区不是正式 Memory，候选生成、暂存和审核不属于 T5 正式写入。候选样本与正式 Memory 必须保持物理或逻辑隔离。任何候选进入正式 Memory，都必须先形成版本化训练批次并完整执行 T0—T8。

推荐第一版链路：

```text
用户提问
→ 检索第1—4级已有知识
→ 生成并执行 SQL
→ 用户明确确认结果正确
→ 形成学习候选
→ 保存到候选区，不写正式 Memory
→ 一个或多个候选组成版本化训练批次
→ 完整执行 T0—T8
→ T5 阶段受控调用 AgentMemory.save_tool_usage()
→ T6—T8 完成写后核验、回归和交付
→ 后续相似问题复用
```

允许一个候选样本构成一个微批次。微批次可以减少等待时间，但不得跳过 T0—T8。

不得采用：

```text
SQL 执行成功
→ 无条件自动写入正式 Memory
```

原因：SQL 可执行不等于业务语义正确。

在线持续学习后续可分为：

1. 用户确认后进入候选区，再按 T0—T8 交付。
2. 人工审核后进入训练批次，再按 T0—T8 交付。
3. 高置信度规则自动批准候选进入批次，但正式写入仍必须保留批次指纹、备份、写后核验、回归、验收和回滚证据。

上线第一版优先采用第 1 种。

## 八、执行边界

Codex 执行本路线时必须遵守：

- 不得跳过当前阶段。
- 不得把基础设施阶段算作正式训练完成。
- 不得把 P0 / P1 / P2 重新定义为顶级训练等级。
- 不得用第三级既有 SQL 样本替代第一级 107 表覆盖。
- 不得在未完成 0B-4 前写正式 Chroma。
- 不得把 SQL 执行成功直接视为可自动学习。
- 在线持续学习不得成为绕过 T0—T8 的写入旁路。
- 候选样本与正式 Memory 必须物理或逻辑隔离。
- 任何 `AgentMemory.save_tool_usage()` 正式写入都只能发生在获批批次的 T5。
- 每次只执行当前明确授权的一个阶段。
- 阶段完成后停止，等待验收再进入下一阶段。
