# AGENTS.md — 水利数据智能问答 Agent

本文件只保留 Codex 协作规则、编码约束、修改流程和项目关键索引。项目架构、启动命令、排障记录和状态说明已拆分到 `docs/`。

## Codex 工作偏好与约束

- 默认用中文回复，代码注释也使用中文。
- 解释保持简洁，避免啰嗦；给出代码改动时说明改了什么、为什么改。
- 用户要求生成的 `.md` 文件或图片，默认输出到 `E:\3\code`；其他类型文件输出前先询问保存位置。
- 不过分夸赞或迎合。回答前反复推敲，优先保证准确性；必要时主动索要补充信息或证据。
- 回答尽量结构化，条理清晰。

## 编码行为准则

### 1. 先思考，再动手

- 不确定时先说明假设并提问，不隐藏困惑。
- 存在多种理解方式时，先列出选项，不擅自替用户决定。
- 如果有更简单的方案，主动说明；该反驳时就反驳。
- 涉及架构选择、版本兼容或明显取舍时，先给出选项和建议，等用户确认。

### 2. 简洁优先

- 用最少的代码解决问题，不写投机性代码。
- 不做超出要求的功能，不为只用一次的代码做抽象。
- 不做未被要求的灵活性、配置项或复杂错误处理。
- 如果实现明显过度复杂，优先重写为更小方案。

### 3. 手术式改动

- 只动必须动的地方，只清理自己造成的问题。
- 不顺手改旁边代码、注释或格式，不重构无关代码。
- 匹配现有代码风格，即使有其他更喜欢的写法。
- 发现无关废弃代码时只提醒，不擅自删除。
- 每一行改动都应能直接追溯到用户需求。

## 代码修改与 Debug 流程

### 一次只改一处

- 一次只修改一个函数、一个路由或一个独立问题点。
- 如果任务涉及多处改动，先列出改动清单和优先级，等用户确认后逐个执行。
- 改动前说明本次“不碰什么”，尤其是不改接口路径、返回结构、数据库连接和前后端契约。

### 每次改动必须可审查

- 改完后展示真实 diff，不能只说“改好了”。
- 涉及多个文件时，每个文件的 diff 分开展示。
- 说明改了什么，具体到函数、模块或关键逻辑。
- 说明为什么改，指出解决的问题。
- 如果借鉴了其他代码或项目，注明来源。

### 每次改动必须可验证

- 给出具体测试方法，具体到可执行命令或操作步骤。
- 如果改变运行时行为，明确说明应观察什么现象来确认成功。
- 修 bug 时优先先复现问题，再修复，再验证。
- 关键操作后用证据确认结果，例如查看进程、健康检查、测试输出或 `git status`。

## 保护接口契约

- 后端 FastAPI 默认端口为 `8000`，前端 Vite 默认端口为 `5173`，前端通过 `/api` 代理到后端。
- 修改前后端任一端时，不随意改动 HTTP 路由路径、SSE/WebSocket 协议、返回 JSON 结构或对方依赖的调用格式。
- 只改内部实现；确实需要变更外部契约时，先说明影响范围并等待确认。

## 项目关键约束

- 项目使用 Vanna 2.0 Agent-based API，不照搬 Vanna 0.x 或错误 quickstart import。
- 排查问答质量问题时，优先检查：检索是否命中、中文 embedding 是否一致、system prompt 是否约束 SQL、SQL 是否排除了 `geom`。
- 排查数据库问题时，先确认 Docker 容器 `local-timescale` 是否运行，以及 `127.0.0.1:5433` 是否可连。
- 排查前端问题时，先确认后端 `/health`，再确认 Vite proxy 到 `http://localhost:8000`。
